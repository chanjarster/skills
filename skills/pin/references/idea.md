# Pin 文档结构定义

## 目录结构

```
pin-docs/
├── 业务知识/
│   ├── 0-索引.md
│   ├── 1-需求文档/
│   │   ├── 0-overview.md
│   │   ├── 1-<业务领域A>.md
│   │   ├── 2-<业务领域B>.md
│   │   └── ...
│   └── 2-操作手册/
│       ├── 1-<业务领域A>.md
│       ├── 2-<业务领域B>.md
│       └── ...
├── 技术文档-X/
│   ├── 0-索引.md
│   ├── 1-技术架构.md
│   ├── 2-开发环境.md
│   └── 3-代码地图/
│       ├── 0-公共组件.md
│       ├── 1-<业务领域A>.md
│       ├── 2-<业务领域B>.md
│       └── ...
├── 规范文档-X/
│   ├── 0-索引.md
│   ├── 1-git规范.md
│   ├── 2-代码规范.md
│   └── 3-示例/
│       ├── 1-<示例A>.md
│       ├── 2-<示例B>.md
│       └── ...
├── 任务日志/
│   ├── 0-索引.md
│   ├── 0-未开始/
│   │   └── <yyMMdd>-<HHmm>-<任务A>/
│   │       ├── task.md
│   │       ├── plan.md
│   │       └── result.md
│   ├── 1-进行中/
│   │   └── <yyMMdd>-<HHmm>-<任务B>/
│   └── 2-已完成/
│       └── <yyMMdd>-<HHmm>-<任务C>/
└── assets/
    └── (图片、截图等资源)
```

## 0-索引.md

索引的目的是便于 AI 精准搜索文档，而不是把所有文档都扫描一遍。

每个大类的文档目录下都有一个 `0-索引.md` 文档，文档内容是该目录下的每个文档的关键词、摘要、路径等信息。

## 业务知识（文件夹）

帮助 AI 从用户的视角了解整个项目。

### 业务知识/1-需求文档

按照领域驱动设计的思想，以业务领域为粒度，以边界上下文（Boundary Context）为边界，来组织文档。

#### 0-overview.md

内容涵盖：
- 整体项目背景
- 业务名词和概念（Ubiquitous language）
- 领域清单，名称、关键词、摘要
- 领域关系图

#### <业务领域X>.md

一个业务领域，一个单独文档。

### 业务知识/2-操作手册

告诉 AI 如何使用系统，从用户的角度了解系统的功能是怎么设计的。

一个业务领域，一个单独文档，和 `业务知识/1-需求文档` 保持一致。

## 技术文档-X（文件夹）

帮助 AI 从代码角度，了解系统是怎么设计的。

可以按照分类，创建多个技术文档目录，比如 "技术文档-前端"、"技术文档-后端"。

### 1-技术架构.md

技术栈，内容：
- 开发语言
- 重要依赖，不要把所有依赖都列出来，因为这些信息都可以从 Maven、npm 的依赖文件里获得
- 其他必要的内容

技术架构，内容应当简要，包括：
- 微服务清单，名称、关键词、摘要
- 模块清单，名称、关键词、摘要
- 微服务之间的关系
- 模块之间的关系

### 2-开发环境.md

内容包括：
- 怎么在本地启动项目
- 怎么做 Debug
- 怎么使用外部资源（比如数据库、中间件等）
- 怎么构建和打包
- 怎么 CI/CD

### 3-代码地图（文件夹）

一个业务领域，一个单独文档，和 `业务知识/1-需求文档` 保持一致。

**尽量精简**，不要把代码中显而易见的信息写到文档里，比如：属性名、字段名、方法名、详细逻辑。

#### 0-公共组件.md

涵盖公共组件、公共领域的代码索引，包括：
- 代码清单，每个代码文件的关键词、摘要、做什么的、文件路径
- 实体之间的关系
- 代码之间的调用关系
- 公共 API 清单，每个 API 的关键词、摘要、做什么的、对应代码文件的路径

#### <业务领域>.md

业务领域的代码索引，包括：
- 代码清单，每个代码文件的关键词、摘要、做什么的、文件路径
- 实体之间的关系
- 代码之间的调用关系
- API 清单，每个 API 的关键词、摘要、做什么的、对应代码文件的路径

## 规范文档-X（目录）

告诉 AI 应当遵循的软件工程规范。

可以按照分类，创建多个规范文档目录，比如 "规范文档-前端"、"规范文档-后端"。

### 1-git 规范.md

告诉 AI 怎么管理 git 代码仓库。内容包括分支规范、commit 规范等。

### 2-代码规范.md

告诉 AI 在编写代码时应当遵循怎样的规范。

文档中包含：
- 总体要求和公共规范
- 代码示例的索引，包括示例名称、关键词、摘要、示例路径。

### 3-示例

通过具体的示例，告诉 AI 在不同情况下应当遵循什么样的规范。

示例应当精简，同时能够在代码里一眼看明白的东西，不要单独写一个示例。

示例可以是：
- 怎么使用框架（不论是开源框架，还是自制框架）
- 某个工具类的用法
- 调用 API 的规范
- 怎么做认证
- UI 配色
- 怎么做单元测试、集成测试、e2e 测试、浏览器测试等
- 复杂的示例甚至可以是一个目录

## 任务日志（文件夹）

管理开发任务日志，帮助 AI 和整个团队了解项目的演变历史。

任务有三种状态：未开始、进行中、已完成，分别对应不同的目录。

每个任务都必须新建一个目录 `<yyMMdd>-<HHmm>-<任务标题>`。

任务目录下有：
- task.md：任务的描述
- plan.md：开发计划、代码规划
- result.md：任务执行的结果

## 文档要求

文档内容应当简明扼要。
凡是表达关系的、流程的图，使用 mermaid 绘制。
